# HR Domain — Final Enterprise Audit & Production Readiness

Snapshot at the end of the fixed 14-module HR rollout (Employee → PayrollItem). Written from a
skeptical, independent-audit posture, not a self-congratulatory summary — every claim below is
backed by a command actually run this session, not assumed.

## 1. Scope Completed

All 14 modules: **Employee, Department, JobTitle, Shift, ShiftSettings (relocated to
`finance/cashier-shift-settings`), AttendanceSettings, AttendanceRecord, EmployeeSettings,
EmployeeFinancialProfile, EmployeeFinancialTransaction, EmployeeAdvance, LeaveRequest,
PayrollSettings, PayrollItem.**

Every module started in one of two states: a hand-written service with method signatures
incompatible with `BaseController` (silently broken beyond create/read — 6 modules: Employee*,
AttendanceRecord*, EmployeeFinancialProfile, EmployeeFinancialTransaction, EmployeeAdvance,
LeaveRequest, PayrollItem — *fixed ad hoc/earlier before formal rollout), or a completely empty
placeholder (AttendanceSettings, PayrollSettings). Every one is now Repository-Pattern-compliant,
RBAC-secured, and has at least one genuine business-logic engine — not a single module in this
rollout is CRUD-only.

## 4. What Is Genuinely Production-Ready vs. What Is Architecture-Only

**Production-ready** (real logic, tested, no known correctness gaps beyond documented limitations):
Employee identity/lifecycle, Department/JobTitle hierarchy, Shift scheduling, AttendanceSettings
policy resolution, AttendanceRecord lateness/overtime computation, EmployeeSettings leave-policy
engine, EmployeeFinancialProfile compensation/eligibility, EmployeeFinancialTransaction ledger +
workflow, EmployeeAdvance full state machine, LeaveRequest full state machine + balance engine,
PayrollSettings policy validation, PayrollItem's Formula Engine itself (evaluator, validator, cycle
detector — all independently correct and tested).

**Architecture-only, explicitly not built** (by design, out of this rollout's scope, documented at
every module that touches it): accounting journal-entry posting (referenced by
`PayrollSettings.accountingIntegration` and `PayrollItem.accounting`, zero posting code exists
anywhere in this project); `hr/payroll`'s actual calculation orchestration (HD-013) — the single
largest remaining gap, since every other module in this rollout exists specifically to feed it data
it does not yet consume.

## 5. Known, Accepted Limitations (not bugs — documented design boundaries)

- Leave balance carry-forward across more than one policy year is not computed (HD-018).
- No scheduled jobs exist for auto-payroll-generation, auto-lock, or auto-complete of leave requests
  — every such transition is a manual API call today.
- No per-branch payroll cycle override, despite the reserved flag (`PayrollSettings.allowBranchCycleOverride`).
- `PayrollItem.isMandatory`/`.isProrated` are classification flags with no enforcement engine yet.
- Multi-currency formula evaluation assumes pre-normalized numbers.
- `FT-001` (bulk-delete route shadowing) and `FT-002` (`BaseController.count()` ignoring filters)
  remain open Foundation-level issues, present in every HR router by the same shared template — not
  specific to this domain, not fixed here per the established "don't fix Foundation issues mid-domain-
  rollout" discipline (except `FT-004`, escalated and fixed given its severity — see
  `BACKEND_FOUNDATION_TECH_DEBT.md`).

## 7. Honest Self-Assessment of This Audit

This audit performed: a full regression suite run, typecheck, lint, live boot, and fresh greps for
structural consistency (mounted routers, RESOURCE_ENUM, no remaining broken services). It did **not**
re-derive every business rule from first principles across all 14 modules in this pass — that
derivation happened at each module's own turn, with its own dedicated tests, and is not re-litigated
here. If a genuine defect is found later, it belongs in `HR_TECHNICAL_DEBT.md` with the same
found-while/fixed-while discipline used throughout this rollout, not silently patched.

## 8. Recommendation

The HR domain is ready for `hr/payroll`'s own dedicated build-out (module 15, next natural step, out
of this rollout's original scope) and for the Foundation-level tech-debt pass (`FT-001`/`FT-002`,
`organization/delivery-area`'s unverified index instance under `FT-003`). Both are cleanly scoped,
already-documented next steps — not blockers discovered by surprise.
