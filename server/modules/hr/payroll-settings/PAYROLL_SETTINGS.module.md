# Payroll Settings (HR) — Engineering Documentation

## 1. Purpose

The Payroll Policy Engine — the single source of truth for how a brand's payroll cycle, approval
workflow, lock/retroactive rules, rounding, tax/insurance defaults, and accounting posting targets
are configured. Module 13, the second-to-last of the fixed 14-module HR rollout.

**This is policy, not calculation.** `PayrollSettings` does not compute a single salary, deduction,
or tax amount itself — it defines the *rules* a future calculation engine (`hr/payroll`, pre-existing
in this codebase, confirmed pure CRUD with zero calculation logic — `HR_TECHNICAL_DEBT.md` HD-013)
would read. Building that calculation engine is explicitly **out of this rollout's scope** — `Payroll`
was never one of the fixed 14 modules; only `PayrollSettings` and `PayrollItem` (module 14, next) are.

**Previously**: a completely empty placeholder — 4 files, each a single comment line, no model, no
router mounted, nothing to review or fix. Built from scratch this turn.

## 2. Architecture

```
PayrollSettings (this module)
    │  policy only — no calculation
    ▼
hr/payroll (pre-existing, CRUD-only, HD-013)
    │  intended future calculation engine — NOT built in this rollout
    ▼
Journal Entries (accounting/) — NOT built; architecture prepared only (§7)
```

Every other financial/HR module this rollout built stays the authority for ITS OWN domain —
`PayrollSettings` does not duplicate any of them:

| Concern | Owned by | `PayrollSettings` role |
|---|---|---|
| Per-employee basic salary, currency, disbursement | `EmployeeFinancialProfile` (module 9) | none — reads nothing from here |
| Individual bonus/deduction/tip transactions | `EmployeeFinancialTransaction` (module 10) | none |
| Salary advances | `EmployeeAdvance` (module 11) | none |
| Leave day entitlement/balance | `EmployeeSettings.leavePolicy` (HD-016) | none |
| Attendance/overtime/late/night-differential policy | `AttendanceSettings` (module 6) | none |
| **Payroll cycle, defaults, automation** | **`PayrollSettings`** (this module) | **authoritative — absorbed from `EmployeeSettings.payroll`, HD-020** |
| Tax/insurance brand-wide default rate | **`PayrollSettings.taxDefaults`** (new) | **authoritative — did not exist anywhere before this turn** |
| Accounting posting target accounts | **`PayrollSettings.accountingIntegration`** (new) | **authoritative — architecture only, no posting engine yet** |

## 3. Workflow / Payroll Lifecycle (policy perspective)

`PayrollSettings` doesn't execute a lifecycle itself — it defines the rules the (unbuilt) calculation
engine would follow:

1. **Cycle**: `cycle.frequency` determines how often a period runs. For monthly: attendance/leave/
   transaction data up to `cycle.cutOffDay` is included; the period is calculated and reviewed by
   `cycle.closingDay`; employees are paid on `cycle.payDay` (which may fall in the next calendar
   month — see §4).
2. **Automation**: if `automation.autoGenerate` is set, a scheduled job (not built) would generate
   draft payroll runs on `automation.autoGenerateDay`.
3. **Approval**: if `approvalWorkflow.requireApproval`, a calculated run needs HR sign-off before
   payment; `requireFinanceApproval` adds a second, finance-level gate (reserved).
4. **Lock**: `lockPolicy.lockAfterDays` (days after `closingDay`) would auto-lock a period against
   further attendance/leave edits — `allowBackdatedAttendance`/`allowBackdatedLeave` control whether
   edits to *already-closed* prior periods are permitted at all, and
   `allowRetroactiveAdjustment`/`retroactiveWindowDays` bound how far back a manual payroll
   adjustment may reach.
5. **Rounding**: `roundingPolicy` would be applied to every computed monetary line before totals are
   finalized.

## 4. Business Rules (fail-fast validation)

All new this turn — the model previously didn't exist.

1. **Coherent cycle** (`assertCoherentCycle`): for non-daily frequencies, `closingDay` cannot be
   before `cutOffDay` (you cannot close a period before its data is cut off). `payDay` cannot equal
   `cutOffDay` or `closingDay` (an unambiguous same-day contradiction) — but `payDay` is **not**
   required to be numerically after `closingDay`, because real payroll routinely pays in the
   following calendar month (e.g. cutOff=25, close=28, pay=5 *of next month* — numerically `5 < 28`
   but chronologically later; the model does not track which month each day belongs to, only that the
   three are distinct).
2. **Coherent approval workflow** (`assertCoherentApprovalWorkflow`): `requireFinanceApproval` cannot
   be `true` while `requireApproval` is `false` — a second-tier gate implies the first tier exists.
3. **Valid accounting integration** (`assertValidAccountingIntegration`): every configured account
   reference must exist in this brand, have `allowPosting: true`, and `status: "active"` — checked
   against `accounting/account` (pre-existing model) at both create and update time.
4. **Retroactive-window consistency** (`pre("save")`): `retroactiveWindowDays` is forced to `0`
   whenever `allowRetroactiveAdjustment` is `false` — same consistency-hook pattern as
   `EmployeeSettings.leavePolicy`'s carry-forward fields.
5. **One settings document per brand** (`{brand}` unique index — same convention as
   `EmployeeSettings`/`AttendanceSettings`).
6. **`update()` re-validates the MERGED document**, not just the partial payload — a client updating
   only `cycle.payDay` is still checked against the *existing* `cutOffDay`/`closingDay` (same pattern
   already used by `EmployeeFinancialProfile`/`AttendanceRecord`'s overridden `update()`).

## 5. Restaurant Examples

- **A single-brand, single-currency casual-dining chain**: `cycle.frequency: "monthly"`,
  `cutOffDay: 25`, `closingDay: 28`, `payDay: 5` (of the following month) — standard monthly payroll
  paid a week into the next month.
- **A cloud-kitchen with daily-wage kitchen staff**: `cycle.frequency: "daily"` — day-of-month cycle
  fields are ignored (§4.1); each shift/day is its own payroll unit (the actual daily-rate
  calculation itself is `EmployeeFinancialProfile.compensation.salaryType: "daily"` + `hr/payroll`'s
  future engine, not this module).
- **A multi-branch brand wanting kitchen staff paid weekly while office staff are paid monthly**: not
  supported today — `allowBranchCycleOverride` is a reserved flag for exactly this (§12), but no
  actual per-branch cycle override is implemented; the whole brand shares one `PayrollSettings`
  document.
- **Ramadan/seasonal schedule changes**: not modeled here at all — `AttendanceSettings.workCalendar`
  (module 6) already owns holiday/working-day policy; `PayrollSettings` only owns the payroll *cycle*
  timing, not the attendance calendar the cycle's calculation would read.

## 6. HR Impact

No previous module's own business logic changed as a result of this module's existence, **except**
`EmployeeFinancialProfile`'s compensation-defaults resolution (§9, HD-020) — reading `PayrollSettings`
now instead of the removed `EmployeeSettings.payroll`. Every other HR module reviewed this turn
(`Employee`, `AttendanceSettings`, `AttendanceRecord`, `EmployeeFinancialTransaction`,
`EmployeeAdvance`, `LeaveRequest`, `JobTitle`, `Department`, `Shift`) needed no change — see §13.

## 7. Accounting Impact (architecture only)

`accountingIntegration` holds 11 optional account references (`salaryExpenseAccount`,
`payrollPayableAccount`, `advanceAccount`, `loanAccount`, `taxPayableAccount`,
`insurancePayableAccount`, `cashAccount`, `bankAccount`, `serviceChargeLiabilityAccount`,
`tipsPayableAccount`, `employerContributionExpenseAccount`) — every one maps onto an existing
`Account.systemRole` value (`PAYROLL_EXPENSE`, `ACCRUED_SALARY`, `EMPLOYEE_ADVANCE`, `TAX_PAYABLE`,
`CASH`, `BANK`, ...). **No journal-entry generation exists anywhere in this project** (deferred
project-wide, consistent with every other module in this rollout) — this is deliberately just the
architecture a future posting engine would read, validated for correctness now (§4.3) so that engine
doesn't have to re-validate account existence/posting-eligibility itself.

## 8. Extension Points

- `automation.autoGenerateDay` → a scheduled job (not built).
- `lockPolicy.lockAfterDays` → an enforcement point in a future `hr/payroll` calculation flow (not
  built) that checks this before accepting attendance/leave edits against a period.
- `accountingIntegration.*` → a future journal-posting engine (not built).
- `allowBranchCycleOverride` → a future per-branch `PayrollSettings` override document (not built;
  currently brand-wide only, same limitation `AttendanceSettings` deliberately avoided by actually
  supporting branch overrides — this module chose not to for scope reasons, see §12).

## 9. API Reference

Base path: **`/api/v1/hr/payroll-settings`**. Standard CRUD + soft-delete/restore/bulk, plus:

- **`GET /resolve`** — `{source: "brand"|"schemaDefault", settings: {...}}`, same pattern as
  `AttendanceSettings`/`EmployeeSettings`.

**Known issue (Foundation, not this module — see `BACKEND_FOUNDATION_TECH_DEBT.md` FT-001):**
`DELETE /bulk-delete` is unreachable (shadowed by `/:id` DELETE).

## 10. Repository

`payroll-settings.repository.js` — `findForBrand(brandId)` (the singleton lookup, same convention as
every other Settings module this rollout built) and `findAccountForScope(accountId, brandId)` (used
by the service's accounting-integration validation).

## 11. Services

`payroll-settings.service.js` — `assertCoherentCycle`, `assertCoherentApprovalWorkflow`,
`assertValidAccountingIntegration`, `validatePolicy` (creation-time, merges onto schema defaults),
and an overridden `update()` (merges onto the existing document before re-validating — §4.6).

## 12. Validation

`payroll-settings.validation.js` — thin: every nested group (`cycle`/`defaults`/`automation`/
`approvalWorkflow`/`lockPolicy`/`roundingPolicy`/`taxDefaults`/`accountingIntegration`) is a plain
nested object, so `joiFactory`'s generic dotted-path reassembly handles all of them correctly with no
manual override needed (unlike `AttendanceSettings`' `holidays`/`breaks`, which are DocumentArrays
and needed one).

## 13. State Flow / Sequence (text)

```
1. Brand admin: POST /hr/payroll-settings {cycle, defaults, ...}
   → service.beforeCreate() → assertCoherentCycle + assertCoherentApprovalWorkflow
     + assertValidAccountingIntegration (if provided) → saved.

2. HR creates an EmployeeFinancialProfile without specifying compensation.salaryType/currency/payDay
   → employee-financial-profile.service.js#beforeCreate calls
     findPayrollSettingsForBrand() → resolveCompensationDefaults()
     → profile inherits the brand's PayrollSettings.defaults/.cycle.payDay.

3. (Not built) hr/payroll's future calculation engine would, for a given period:
   a. Read PayrollSettings for cycle/rounding/lock/tax-default policy.
   b. Read EmployeeFinancialProfile for each employee's basicSalary/currency/eligibility.
   c. Read AttendanceRecord for worked/late/overtime minutes (module 7).
   d. Read LeaveRequest for approved-leave payroll effects (module 12, HD-018).
   e. Read EmployeeAdvance for due installments (module 11, HD-015).
   f. Read EmployeeFinancialTransaction for bonuses/deductions/tips this period (module 10).
   g. Sum, round per PayrollSettings.roundingPolicy, and produce a Payroll document.
   h. (Future) post a balanced Journal Entry using PayrollSettings.accountingIntegration.
```

## 17. Examples

**Minimal valid document** (everything else defaults):
```json
{ "brand": "<brandId>" }
```
Resolves to: monthly cycle, cutOff 25 / close 28 / pay 1, EGP currency, approval required, no
accounting integration configured, rounding to nearest 2 decimals.

**A brand wanting no approval gate and quarter-precision rounding**:
```json
{
  "brand": "<brandId>",
  "approvalWorkflow": { "requireApproval": false },
  "roundingPolicy": { "method": "nearest", "precision": 2 }
}
```

## 18. Common Mistakes

- **Setting `payDay` equal to `cutOffDay` or `closingDay`** thinking "pay immediately at close" — the
  model rejects this as ambiguous; use a `payDay` numerically different even if conceptually
  "the next available date."
- **Enabling `requireFinanceApproval` without `requireApproval`** — rejected; enable both.
- **Referencing an inactive or posting-disabled `Account`** in `accountingIntegration` — rejected at
  write time with a clear per-field error, not a silent no-op.
- **Assuming this module calculates anything** — it is pure policy; look at `hr/payroll` (unbuilt
  calculation engine) for where the numbers would actually be produced.

## 19. Best Practices

- Treat `PayrollSettings` as the *only* place payroll cycle/defaults/tax-default/accounting-target
  policy lives — do not add a parallel field to `EmployeeSettings` or `EmployeeFinancialProfile` for
  anything already here (that was exactly HD-020's mistake, now corrected).
- When `hr/payroll`'s real calculation engine is eventually built, it should read this module's
  policy fields directly rather than re-deriving equivalent constants — every field here exists
  specifically to avoid that duplication.
- Keep `accountingIntegration` references validated at write time (already enforced) rather than
  discovering a broken/inactive account reference at journal-posting time, once that engine exists.
